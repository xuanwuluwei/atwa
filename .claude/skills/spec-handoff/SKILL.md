# Spec + Handoff Skill

## 两个文档的职责边界

理解这一点是用好此 Skill 的前提：

| | **Spec** | **Handoff Note** |
|---|---|---|
| 是什么 | 这个需求"现在是什么样的" | 这次交付"发生了什么" |
| 生命周期 | 长期存在，持续更新 | 跟着一次交付走，完成后归档 |
| 内容原则 | 永远反映当前真实状态 | 只记录"Spec 里看不出来"的东西 |
| Git 角色 | 活文档 | 历史记录 |

**判断写哪里的简单规则：**
- 这件事以后还需要知道 → 写进 Spec
- 这件事只在这次交付过程中重要 → 写进 Handoff Note
- 两者都适用 → Spec 写结论，Handoff Note 写过程和原因

## 文件组织

```
docs/
├── specs/
│   ├── user-list.md
│   └── approval-flow.md
└── handoff/
    ├── 2024-03-user-list.md
    └── 2024-04-approval-flow.md        ← 第一次交付
    └── 2024-06-approval-flow-delegate.md  ← 后续迭代，新建文件
```

**命名规则：**
- Spec：`docs/specs/<需求关键词-kebab-case>.md`
- Handoff：`docs/handoff/<YYYY-MM>-<与对应spec相同的关键词>.md`
- 多轮迭代时在末尾加描述词区分：`-delegate`、`-v2`、`-refactor`

## 第一步：判断当前场景

收到请求后，先判断属于哪种场景，再执行对应动作：

| 场景 | 判断依据 | 动作 |
|---|---|---|
| **全新需求** | 没有对应 spec 文件 | 新建 Spec + 新建 Handoff Note |
| **开发中更新** | 用户 @ 了某个 spec，描述进展或变更 | 仅更新 Handoff Note Dev log |
| **需求变更（开发中）** | 需求改了，但还没上线 | 更新 Spec + 在当前 Handoff Note Dev log 追加 |
| **需求变更（已上线）** | 需求改了，上一版已上线 | 更新 Spec + 新建 Handoff Note，注明变更来源 |
| **归档** | 用户说上线了、完成了 | 更新 Handoff Note 状态为已归档，确认 Spec 是最新的 |
| **查询进度** | 用户问某需求状态 | 读取对应文档，输出摘要 |

**如果场景不明确，先问用户，不要猜。**

---

## 第二步：定位文档

- 用户 **@ 了 spec 文件** → 以该文件为锚点，根据文件名推导对应的 Handoff Note
- 用户**描述了需求内容** → 在 `docs/specs/` 下查找匹配文件名
- **找到多个候选文件** → 列出来让用户确认，不要自己猜
- **找不到对应文件** → 询问用户是否是新需求，确认后再新建

---

## 第三步：执行动作

### 新建需求

**信息不足时主动追问，不要脑补填空。** 至少需要确认：
1. 需求的核心目标是什么
2. 边界是什么（不做什么）
3. 有没有已知的约束或特殊逻辑

追问不超过 3 个问题，收到回答后再生成。

生成后说明：哪些字段是根据描述填写的，哪些是空白需要用户补充。

### 更新文档

- Dev log **只追加，不改写**——保留决策轨迹
- Spec 有变更时，同步在 Decision log 里加一条记录
- 不要在没有用户指令的情况下删除任何内容

### 归档

归档时做两件事：
1. Handoff Note 状态改为 `已归档`，加上归档日期
2. 检查 Spec 是否反映了最终上线的状态，如有出入提醒用户

```markdown
**PR / 分支**：

---

## 背景
> 为什么现在做这个，一两句就够。

## 本次交付范围
- 新增：
- 修改：
- 删除：

## 重点说明
> 只写 Spec 里没有的东西：
> - 和其他团队/接口对齐的结论
> - 临时方案或已知遗留问题
> - "我们决定不做 X，因为……"

> 只追加，不改写。记录过程中的决策、变更、阻塞。

- [ ] 核心流程端到端可用
- [ ] Spec 中的边界状态均已处理
- [ ] 错误状态有合理的降级处理
- [ ] 相关功能无回归
- [ ] （根据需求补充具体检查项）
```

## 需求变更处理规则

**变更发生在开发中（未上线）：**
1. 直接修改 Spec，更新为新的真实状态
2. 在 Spec Decision log 追加：`[日期] 变更：X 改为 Y，原因是……`
3. 在当前 Handoff Note Dev log 追加：`[日期] 需求变更，Spec 已同步更新，影响范围：……`

**变更发生在上线之后：**
1. 直接修改 Spec（不新建 spec 文件）
2. 在 Spec Decision log 追加变更记录
3. 新建 Handoff Note，在"变更自"字段注明上一个 Handoff

**永远不要：**
- 创建 `spec-v2.md` 或 `spec-old.md`（用 Git 历史保留旧版本）
- 用新内容覆盖 Dev log 里的旧记录
- 在 Spec 里记录过程信息（过程信息属于 Handoff Note）

---

## 防止文档腐化

以下情况主动提醒用户：

- 用户描述的改动和当前 Spec 内容有出入时：指出差异，问是否需要同步更新
- 用户说"上线了"但 Handoff Note 状态还是进行中时：提示归档
- 用户 @ 了一个 Spec，但对应的 Handoff Note 超过 30 天没有更新时：提示确认状态是否仍准确

---

## 输出要求

- 始终创建实际文件，不在对话里展示内容了事
- 如果 `docs/specs/` 或 `docs/handoff/` 目录不存在，自动创建
- 生成后告知：文件路径、哪些字段需要用户补充
- 字段空白时用 `<!-- 待补充：xxx -->` 标注，不要填占位废话

---

## 每次代码改动结束前的检查

在回复用户"完成了"之前，必须先问自己：
- 这次改动有没有对应的 Handoff Note？
- Dev log 有没有记录刚才做的事？
- 如果没有，现在补上，再结束。
