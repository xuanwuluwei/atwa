## 6. 常见问题排查指引

> 本章节按症状分类，为模型提供从错误现象到根因的完整排查路径。

---

### 6.1 tmux 相关问题

**症状**：`discover_all_panes()` 返回空列表，但手动执行 `tmux ls` 有输出

**排查路径**：
1. 检查测试进程的环境变量：`print(os.environ.get("ATWA_TMUX_SOCKET"))`
2. 检查 libtmux 指向的 socket：`libtmux.Server().socket_path`，确认与实际 tmux 使用的 socket 一致
3. 若使用了多个 tmux server（`tmux -L <name>`），确认 `ATWA_TMUX_SOCKET` 与实际 socket 名称匹配
4. 尝试直接调用验证 PATH：`subprocess.run(["tmux", "ls"], capture_output=True).returncode`，非 0 则 tmux 不在 PATH

**症状**：`tmux send-keys` 命令返回成功但 pane 没有收到输入

**排查路径**：
1. 确认 `pane_id` 格式正确（`%数字`，如 `%3`），用 `tmux display-message -t %3 -p "#{pane_id}"` 验证 pane 存在
2. 确认 pane 不处于 vi/emacs/less 等 alternate screen 模式，这些模式下键盘映射不同
3. 确认 `send-keys` 命令的 `-t` 参数使用 pane_id 格式而非 `session:window.pane` 格式
4. 在测试 pane 中手动执行 `tmux send-keys -t %<id> "echo test" Enter` 验证是否有效

### 6.2 Event Parser 问题

**症状**：`waiting_input` 未被检测到，pane 卡在等待输入状态但 status 仍为 `active`

**排查路径**：
1. 打印 `clean_ansi_output()` 的返回值，确认 ANSI 码已被清除，prompt 文本可见
2. 确认 prompt 文本在 `SCROLLBACK_LINES` 范围内——若 agent 输出太长（超过 200 行），早期内容会被截断
3. 将实际 agent 输出保存到 `output_samples/` 目录，加入 TC-PARSE-002 的参数化测试，确认是否匹配现有 pattern
4. 若是新格式，在 `WAITING_INPUT_PATTERNS` 列表中添加对应正则表达式

**症状**：stuck 检测误报（正常运行时触发）

**排查路径**：
1. 打印触发 stuck 的 5 段输出内容，检查是否真的高度相似，或是 agent 的固定 header/footer 导致相似度虚高
2. 调高 `config.toml` 中的 `stuck_similarity` 阈值（0.92 → 0.96）
3. 在 `StuckDetector` 中添加忽略规则：对纯进度条格式的输出（如 `[=====>  ] 60%`）跳过相似度计算
4. 检查 `window_size` 配置，若设置太小（< 5），少量相似输出就会误触发

---

### 6.3 FastAPI 和 WebSocket 问题

**症状**：WebSocket 连接建立后没有收到初始状态推送

**排查路径**：
1. 检查 server 的 `on_connect` 事件处理器是否调用了 `broadcast_initial_state(websocket)`
2. 确认数据库中有数据（空库时初始推送应为空数组 `[]`，而非 `None` 或不发送）
3. 检查 WebSocket 路由 `/ws/sessions` 是否正确注册在 FastAPI app 上
4. 查看 server 日志是否有 `INFO: WebSocket connected` 记录，若没有则连接未到达 handler

**症状**：PATCH /api/sessions 返回 200 但数据未持久化（刷新后丢失）

**排查路径**：
1. 检查 handler 中是否执行了 `await db.commit()`——SQLAlchemy async 不会自动提交事务
2. 查看 server 日志中是否有 `WARNING` 或 `ERROR`，可能存在事务回滚
3. 确认测试数据库路径与 server 使用的路径一致（对比 `ATWA_DB_PATH` 和 `db.url`）
4. 用 `sqlite3 /tmp/atwa_test.db "SELECT display_name FROM pane_sessions"` 直接查询数据库验证

### 6.4 Playwright E2E 问题

**症状**：`[data-testid="session-card"]` 找不到元素，测试报 TimeoutError

**排查路径**：
1. 截图检查实际 UI 状态：在失败的测试步骤前加 `await page.screenshot({ path: 'debug.png' })`
2. 确认前端组件中存在 `data-testid` 属性（重构时容易被误删）
3. 检查 `playwright.config.ts` 中的 `webServer` 是否成功启动（增加 `timeout: 30000` 给更多时间）
4. 打开 `headless: false` 模式人工观察页面行为，确认是渲染问题还是数据问题

**症状**：TC-E2E-RT-002（状态更新延迟）偶发性失败

**排查路径**：
1. 检查 CI 环境资源是否充足（CPU 限制会导致 daemon 轮询延迟）
2. 将 `timeout` 从 1000ms 放宽到 2000ms，并在测试中记录实际延迟值用于分析
3. 确认 `CAPTURE_INTERVAL_ACTIVE_MS` 在测试环境中为 500ms（若被设置为更大值则延迟增加）
4. 检查 WebSocket 广播逻辑是否在 daemon 主线程中被 DB 写入操作阻塞（应使用 `asyncio.create_task` 异步广播）

**症状**：TC-E2E-FOCUS-001（iTerm2 跳转）在 CI 中失败

**排查路径**：
1. CI 环境无图形界面，AppleScript `activate` 必定失败
2. 确认 CI 的环境变量中已设置 `ATWA_SKIP_ITERM2=1`
3. 若需在 CI 中验证调用链，改为使用 `page.waitForRequest()` 拦截 API 请求而非验证实际窗口切换效果
4. 本地测试失败时：确认 iTerm2 已在「系统设置 → 隐私与安全 → 辅助功能」中授权

### 6.5 数据库并发问题

**症状**：`sqlite3.OperationalError: database is locked`，集中出现在并发测试中

**排查路径**：
1. 确认 SQLite 已开启 WAL 模式：连接后执行 `PRAGMA journal_mode=WAL`，应返回 `wal`
2. 确认所有连接使用了超时参数：`sqlite3.connect(db_path, timeout=30)`
3. 检查是否有连接泄漏——确保所有数据库操作使用 `async with get_db() as db:` 而非手动管理连接
4. 在 SQLAlchemy 连接池配置中添加 `pool_pre_ping=True` 和 `pool_timeout=30`
5. 若问题持续，将 TC-DB-002 的并发数从 50 降到 20，确认是 SQLite 能力边界还是代码问题
