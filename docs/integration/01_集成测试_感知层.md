# ATWA 集成测试 — 01 感知层

> 覆盖模块：tmux Discovery、Output Capture、Event Parser
---

## 3.1 tmux Discovery 模块

**模块职责**：发现当前 tmux server 下所有 session、window、pane，返回结构化的 `PaneInfo` 列表。

---

#### TC-DISC-001：基础枚举——多 session 多 window

**目的**：验证 `discover_all_panes()` 能够正确枚举跨 session、跨 window 的所有 pane。

**前置条件**：
- 测试 tmux 环境已初始化（`atwa_test_main`、`atwa_test_agents` 两个 session 存在）
- `atwa_test_agents` 有 3 个 window，每个 window 1 个 pane

**测试步骤**：
1. 调用 `discover_all_panes()`
2. 过滤出 session 名以 `atwa_test` 开头的结果
3. 统计返回的 pane 数量

**预期结果**：
- 返回列表长度 >= 4（`atwa_test_main` 1个 + `atwa_test_agents` 3个）
- 每个 `PaneInfo` 对象包含字段：`pane_id`、`session_name`、`window_index`、`pane_index`、`is_active`、`pid`
- `pane_id` 格式为 `%数字`（如 `%3`、`%12`）
- `pid` 为正整数

**失败排查**：
- 若返回空列表：检查 `tmux ls` 命令是否能在测试进程的 PATH 中找到 tmux；检查 `ATWA_TMUX_SOCKET` 环境变量是否与实际 tmux socket 匹配
- 若 `pane_id` 格式异常：libtmux 版本问题，检查 `libtmux.__version__` 是否 >= 0.28

---

#### TC-DISC-002：tmux 未运行时的降级处理

**目的**：验证当 tmux server 不存在时，函数返回空列表而非抛出异常。

**前置条件**：使用一个不存在的 tmux socket 名称。

**测试步骤**：
```python
import os
os.environ["ATWA_TMUX_SOCKET"] = "nonexistent_socket_12345"
result = discover_all_panes()
```

**预期结果**：
- 返回 `[]`（空列表）
- 不抛出任何异常
- 日志中有 `WARNING` 级别的记录，内容包含 "tmux server not found" 或类似描述

**失败排查**：
- 若抛出 `libtmux.exc.LibTmuxException`：需要在 `discover_all_panes()` 中增加 try/except 捕获
- 若抛出 `FileNotFoundError`：tmux 不在 PATH 中，检查测试环境的 shell 配置

---

#### TC-DISC-003：pane_id 稳定性验证

**目的**：验证同一个 pane 在多次调用之间返回相同的 `pane_id`。

**测试步骤**：
1. 第一次调用 `discover_all_panes()`，记录所有 `pane_id`
2. 等待 500ms
3. 第二次调用 `discover_all_panes()`，记录所有 `pane_id`
4. 对比两次结果

**预期结果**：
- 两次返回的 `pane_id` 集合完全一致（`set(first_ids) == set(second_ids)`）
- 顺序可以不同，但 id 必须稳定

**失败排查**：
- tmux pane_id 由 tmux server 分配，只要 pane 没有关闭就不会变化；若出现变化说明 fixture 中有 pane 被意外关闭和重建

---

#### TC-DISC-004：pane 关闭后自动消失

**目的**：验证当一个 tmux pane 被关闭后，下次调用不再返回该 pane。

**测试步骤**：
1. 记录初始 pane 列表，取最后一个 `pane_id` 作为目标
2. 执行 `tmux kill-pane -t <pane_id>`
3. 等待 200ms
4. 再次调用 `discover_all_panes()`

**预期结果**：
- 新列表中不包含已关闭的 `pane_id`
- 其他 pane 的 `pane_id` 不变

---

## 3.2 Output Capture 模块

**模块职责**：通过 `capture-pane` 和 PTY log 两种模式获取 tmux pane 的文本输出，并通过 pyte 清洗 ANSI 转义码。

---

#### TC-CAP-001：capture-pane 基础抓取

**目的**：验证可以从活跃 pane 抓取到最新输出内容。

**前置条件**：在测试 pane 中执行 `echo "ATWA_TEST_MARKER_12345"`。

**测试步骤**：
1. 向 `atwa_test_main:0.0` pane 发送命令 `echo "ATWA_TEST_MARKER_12345"`
2. 等待 300ms
3. 调用 `capture_pane(pane_id)` 获取输出

**预期结果**：
- 返回的字符串包含 `"ATWA_TEST_MARKER_12345"`
- 不包含 ANSI 转义序列（无 `\x1b[` 字符）

**失败排查**：
- 若返回空字符串：检查 `SCROLLBACK_LINES` 配置是否足够；检查 pane 是否处于 alternate screen 模式（vim/less 等会切换 screen）
- 若包含 ANSI 码：pyte 处理流程未正确调用，检查 `capture.py` 中 `_clean_output()` 函数

---

#### TC-CAP-002：ANSI 转义码清洗验证

**目的**：验证 pyte 能够正确处理包含颜色、光标移动等 ANSI 序列的输出。

**测试步骤**：
```python
# 使用包含 ANSI 序列的样本文本（模拟 Claude CLI 的彩色输出）
raw_output = "\x1b[32m✓ Tool executed\x1b[0m\nBash result: success\x1b[1m (3 files)\x1b[0m"
clean = clean_ansi_output(raw_output)
```

**预期结果**：
- `clean` 等于 `"✓ Tool executed\nBash result: success (3 files)"`
- 不包含任何 `\x1b` 字符
- 换行符保留

**失败排查**：
- pyte 对某些私有 ANSI 序列处理不完整，可在 `_clean_output()` 中增加正则兜底：`re.sub(r'\x1b\[[0-9;]*[mGKHF]', '', text)`

---

#### TC-CAP-003：PTY log 模式读取

**目的**：验证通过 `atwa-wrap` 启动的进程，其 log 文件可以被正确读取和解析。

**前置条件**：
- 使用 `atwa-wrap bash -c 'echo PTY_LOG_TEST; sleep 0.5; echo PTY_DONE'` 启动一个短暂进程
- 等待进程结束

**测试步骤**：
1. 获取该 pane 对应的 log 文件路径 `~/Library/Logs/atwa/pane-<id>.log`
2. 调用 `read_pty_log(pane_id)` 读取内容

**预期结果**：
- 返回内容包含 `"PTY_LOG_TEST"` 和 `"PTY_DONE"`
- 输出顺序正确（PTY_LOG_TEST 在前）
- 文件大小 <= 10MB（超出则触发 rotation）

---

#### TC-CAP-004：轮询频率自适应

**目的**：验证活跃 pane（有输出）使用 500ms 间隔，空闲 pane 使用 3000ms 间隔。

**测试步骤**：
1. 监控一个活跃输出的 pane，记录 10 次 capture 的时间戳
2. 停止输出，等待 pane 进入 idle 状态
3. 记录下 5 次 capture 的时间戳
4. 计算平均间隔

**预期结果**：
- 活跃状态平均间隔：400ms ~ 600ms
- 空闲状态平均间隔：2800ms ~ 3200ms

## 3.3 Event Parser 模块

**模块职责**：将清洁文本解析为结构化的 `ParsedEvent` 列表，识别工具调用、错误、等待输入、完成信号等事件。

> **说明**：Event Parser 是纯函数模块，所有测试使用 `fixtures/output_samples/` 中的文本文件作为输入，不依赖真实 tmux 环境。

---

#### TC-PARSE-001：waiting_input 检测——[y/n] 格式

**目的**：验证最常见的 `[y/n]` 格式的等待输入提示被正确识别。

**输入样本**（`claude_waiting_yn.txt`）：
```
Analyzing codebase...
Found 3 potential issues in routes.py:
  - Missing error handler on line 45
  - Unused import on line 12
  - Type mismatch on line 78
Continue with fixes anyway? [y/n]
```

**测试步骤**：
```python
text = load_fixture("claude_waiting_yn.txt")
events = parse_output(pane_id="%1", text=text)
waiting_events = [e for e in events if e["event_type"] == "waiting_input"]
```

**预期结果**：
- `waiting_events` 长度为 1
- `waiting_events[0]["data"]["prompt"]` 包含 `"Continue with fixes anyway? [y/n]"`
- `waiting_events[0]["confidence"]` >= 0.9

---

#### TC-PARSE-002：waiting_input 检测——多种格式覆盖

**目的**：验证所有已知的等待输入格式均能被识别。

**测试用例参数化**：

| 输入文本 | 预期识别 | 说明 |
|----------|----------|------|
| `"Overwrite existing file? (y/N)"` | ✅ | 括号格式，大写 N |
| `"Continue? [Y/n]"` | ✅ | 大写 Y |
| `"Input required: enter your API key"` | ✅ | 自由文本提示 |
| `"Enter your choice (1-3):"` | ✅ | 数字选择 |
| `"Press any key to continue..."` | ✅ | 任意键提示 |
| `"Processing files..."` | ❌ | 不应识别为等待输入 |
| `"[INFO] Task started"` | ❌ | 方括号但非输入提示 |

**预期结果**：前 5 条产生 `waiting_input` 事件，后 2 条不产生。

---

#### TC-PARSE-003：工具调用识别——Claude CLI 格式

**目的**：验证 Claude CLI 的工具调用输出被正确解析为 `tool_start` 和 `tool_end` 事件。

**输入样本**（`claude_tool_bash.txt`）：
```
I'll run the tests to check the current state.

Tool: Bash
Command: cd /project && python -m pytest tests/ -v
─────────────────────────────────────────
...running...
─────────────────────────────────────────
Output:
collected 24 items
test_api.py::test_login PASSED
24 passed in 3.21s
```

**预期结果**：
- 解析出 `tool_start` 事件：`data.tool_name == "Bash"`
- 解析出 `tool_end` 事件：`data.status == "success"`
- 两个事件的 `pane_id` 相同

---

#### TC-PARSE-004：stuck 检测

**目的**：验证重复内容的检测逻辑（使用 textdistance 相似度算法）。

**输入构造**：
```python
repeated_outputs = [
    "Retrying connection to server...\nAttempt 1/10",
    "Retrying connection to server...\nAttempt 2/10",
    "Retrying connection to server...\nAttempt 3/10",
    "Retrying connection to server...\nAttempt 4/10",
    "Retrying connection to server...\nAttempt 5/10",
]
```

**测试步骤**：
```python
detector = StuckDetector(threshold=0.92, window_size=5)
for output in repeated_outputs:
    result = detector.update(pane_id="%1", text=output)
# 最后一次 update 应该触发 stuck
```

**预期结果**：
- 前 4 次 `update()` 返回 `None`（未触发）
- 第 5 次 `update()` 返回 `ParsedEvent`，`event_type == "stuck"`
- `confidence` >= 0.85

**反例验证**：
```python
# 正常进展的输出不应触发
normal_outputs = [
    "Reading file: config.py",
    "Reading file: models.py",
    "Writing updated routes.py",
    "Running tests...",
    "All tests passed",
]
# 所有 update() 均返回 None
```

#### TC-PARSE-006：Parser 无状态验证

**目的**：验证 `parse_output()` 是纯函数，两次相同输入产生相同输出。

**测试步骤**：
```python
text = load_fixture("claude_tool_bash.txt")
result1 = parse_output("%1", text)
result2 = parse_output("%1", text)
assert result1 == result2
```

**预期结果**：两次结果完全相同（字段值和顺序）。

---

#### TC-PARSE-007：空输入处理

**目的**：验证空字符串和纯空白输入不会引发异常。

**测试步骤**：
```python
assert parse_output("%1", "") == []
assert parse_output("%1", "   \n  \t  ") == []
assert parse_output("%1", None) == []
```

**预期结果**：所有情况均返回空列表，无异常。
