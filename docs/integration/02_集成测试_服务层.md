## 3.4 Session Tracker 模块

**模块职责**：维护所有 pane 的状态，根据解析事件和时间阈值驱动状态机转换，将变更写入数据库。

---

#### TC-TRACK-001：状态转换——active → waiting_input

**目的**：验证收到 `waiting_input` 事件后状态正确转换并写入数据库。

**测试步骤**：
```python
tracker = SessionTracker(db=test_db)
tracker.upsert_pane(pane_id="%1", session_name="test", window_index=0, pane_index=0)
tracker.update_status("%1", "active")

event = ParsedEvent(pane_id="%1", event_type="waiting_input",
                    timestamp=now_ms(), data={"prompt": "Continue? [y/n]"}, confidence=0.95)
tracker.process_event(event)
```

**预期结果**：
- `tracker.get_status("%1") == "waiting_input"`
- 数据库 `pane_sessions` 表中该 pane 的 `status == "waiting_input"`
- `status_reason` 包含 prompt 文本片段
- 状态写入数据库的延迟 <= 100ms

---

#### TC-TRACK-002：状态转换——idle_running → stuck（时间驱动）

**目的**：验证超过 `STUCK_THRESHOLD_S` 且内容重复时自动转换为 stuck 状态。

**测试步骤**：
1. 设置 pane 状态为 `idle_running`，`last_output_at` 为 130 秒前
2. 向 stuck detector 注入 5 条高相似度输出
3. 调用 `tracker.tick()` 触发时间检查

**预期结果**：
- 状态转换为 `stuck`
- 数据库记录更新

---

#### TC-TRACK-003：多 pane 优先级排序

**目的**：验证 `get_attention_queue()` 按优先级正确排序需要处理的 pane。

**前置条件**：
- pane `%1`：状态 `retry_loop`
- pane `%2`：状态 `waiting_input`
- pane `%3`：状态 `error_stopped`
- pane `%4`：状态 `active`

**测试步骤**：
```python
queue = tracker.get_attention_queue()
attention_panes = [p for p in queue if p["needs_attention"]]
```

**预期结果**：
- 排序为：`%2`（waiting_input）→ `%3`（error_stopped）→ `%1`（retry_loop）
- `%4` 不出现在 attention queue 中

---

#### TC-TRACK-004：状态转换幂等性

**目的**：验证对同一 pane 重复处理相同事件不会产生副作用。

**测试步骤**：
```python
event = make_waiting_input_event("%1")
tracker.process_event(event)
tracker.process_event(event)  # 重复处理
tracker.process_event(event)  # 再次重复
```

**预期结果**：
- 数据库中只有一条状态变更记录
- `tracker.get_status("%1")` 依然为 `"waiting_input"`（不变）
- 无重复的 `tool_events` 记录

---

#### TC-TRACK-005：pane 关闭自动标记 terminated

**目的**：验证 tmux pane 被关闭后，Tracker 在下一次 tick 中将其标记为 terminated。

**测试步骤**：
1. 在 tracker 中注册 pane `%99`
2. 在 tmux 中关闭该 pane（`tmux kill-pane -t %99`）
3. 等待 2 个 poll 周期（约 6 秒）
4. 检查 tracker 状态

**预期结果**：
- `tracker.get_status("%99") == "terminated"`
- 数据库中 `ended_at` 字段已填充

## 3.5 Attention Tracker 模块

**模块职责**：记录用户在各 tmux pane 间的焦点切换，写入 `attention_log` 表，为 Phase 3 Insight Engine 提供认知负载数据。

---

#### TC-ATTN-001：焦点变化记录

**目的**：验证用户切换 tmux 窗口时，attention_log 正确记录切换事件。

**测试步骤**：
1. 当前焦点在 `atwa_test_main:0`
2. 执行 `tmux select-window -t atwa_test_agents:1`
3. 等待 1.5 秒（超过 1 次轮询周期）
4. 查询 `attention_log` 表

**预期结果**：
- 存在一条 `attention_log` 记录，`pane_id` 对应 `atwa_test_main:0` 的 pane
- `ended_at` 已填充，`duration_ms` 合理（>= 0）
- 存在新的记录指向 `atwa_test_agents:1` 的 pane，`ended_at` 为 NULL（当前焦点）

## 3.6 FastAPI Server 模块

**前置条件**：所有 API 集成测试使用 `httpx.AsyncClient` 以 ASGI 模式直接调用，不需要启动真实 HTTP server。

```python
@pytest.fixture
async def client():
    async with AsyncClient(app=app, base_url="http://test") as c:
        yield c
```

## 3.7 WebSocket 实时推送

---

#### TC-WS-001：连接建立与初始数据推送

**目的**：验证 WebSocket 连接建立后立即推送当前所有 session 状态。

**测试步骤**：
```python
async with websockets.connect("ws://localhost:8743/ws/sessions") as ws:
    message = await asyncio.wait_for(ws.recv(), timeout=2.0)
    data = json.loads(message)
```

**预期结果**：
- 连接建立后 2 秒内收到消息
- 消息 `type == "initial_state"`
- `data.sessions` 数组包含所有当前 pane

---

#### TC-WS-002：状态变更实时推送

**目的**：验证 pane 状态变更后，WebSocket 客户端在 1 秒内收到推送。

**测试步骤**：
1. 建立 WebSocket 连接
2. 通过 API 模拟触发 pane `%1` 状态变更为 `waiting_input`
3. 等待 WebSocket 消息

**预期结果**：
- 1 秒内收到 `type == "session_update"` 消息
- `data.pane_id == "%1"`
- `data.status == "waiting_input"`

---

#### TC-WS-003：无客户端时不阻塞

**目的**：验证没有 WebSocket 客户端连接时，广播操作不会阻塞 daemon 主循环。

**测试步骤**：
```python
broadcaster = WebSocketBroadcaster()
start = time.time()
await broadcaster.broadcast({"type": "session_update", "pane_id": "%1"})
elapsed = time.time() - start
```

**预期结果**：
- `elapsed < 0.01`（广播操作几乎立即返回）
- 无异常抛出

## 3.8 数据库持久化

---

#### TC-DB-001：daemon 重启后数据不丢失

**目的**：验证 pane metadata（display_name、description、tags）在 daemon 重启后依然存在。

**测试步骤**：
1. 写入 pane metadata：`display_name = "重要任务"`
2. 重启 daemon（关闭并重新初始化 SessionTracker）
3. 读取该 pane 的 metadata

**预期结果**：
- `display_name` 依然为 `"重要任务"`
- `description` 和 `tags` 也完整保留

---

#### TC-DB-002：并发写入安全性

**目的**：验证 daemon 和 FastAPI server 并发写入数据库时不出现数据损坏。

**测试步骤**：
```python
# 同时启动 50 个并发写入任务
tasks = [write_tool_event(db, pane_id="%1") for _ in range(50)]
await asyncio.gather(*tasks)

count = await db.scalar("SELECT COUNT(*) FROM tool_events WHERE pane_id = '%1'")
```

**预期结果**：
- `count == 50`（无丢失）
- 无 `sqlite3.OperationalError: database is locked` 异常
- 所有记录字段完整，无 NULL 异常
