## 1. runtime_info 对象

每个 API 响应和 WebSocket 推送中的 `runtime_info` 字段均遵循以下结构。所有字段均由 `daemon/session_tracker.py` 实时计算，通过 `FastAPI → WebSocket → 前端` 链路传递。

```typescript
interface RuntimeInfo {
  // ── 时间维度 ──────────────────────────────────────
  total_elapsed_ms:        number   // session 启动到现在的总耗时（ms）
  current_tool_elapsed_ms: number   // 当前工具调用已耗时（空闲时为 0）
  last_output_ago_ms:      number   // 距上次有输出的时间（ms）

  // ── 当前活动 ─────────────────────────────────────
  current_tool:            string | null  // "Bash" | "Write" | null
  current_step:            number   // 本 session 内工具调用总次数
  thinking:                boolean  // LLM 推理中（有输出但无工具调用）

  // ── 文件操作统计 ──────────────────────────────────
  files_read:              number
  files_written:           number
  files_modified:          number

  // ── Token & 费用 ─────────────────────────────────
  token_input:             number
  token_output:            number
  cost_usd:                number
  context_window_pct:      number   // 上下文窗口使用率（0.0 - 1.0）

  // ── 错误信息 ─────────────────────────────────────
  last_error_summary:      string | null  // 最近一次错误的前 200 字符
  consecutive_errors:      number   // 连续错误次数

  // ── 进程资源 ─────────────────────────────────────
  cpu_pct:                 number   // pane 对应进程的 CPU 使用率
  mem_mb:                  number   // 内存使用（MB）

  // ── 历史对比（Phase 2+ 才有值）────────────────────
  historical_avg_ms:       number | null  // 同类 session 的平均耗时
  vs_historical:           number | null  // 当前 / 历史均值的比率（>1 表示比平均慢）
}
```

### 1.1 计算规范

| 字段 | 计算来源 |
|------|---------|
| `total_elapsed_ms` | `now_ms() - session.started_at` |
| `current_tool_elapsed_ms` | 当前 tool_event 的 `now_ms() - started_at` |
| `context_window_pct` | `(token_input + token_output) / model_max_tokens`，模型上限从配置读取 |
| `cpu_pct` / `mem_mb` | `psutil.Process(pane.pid)` |
| `vs_historical` | `total_elapsed_ms / historical_avg_ms`，historical 来自 session_summaries 均值 |

## 2. Harness 工程规范

ATWA 设计为可被 coding agent 增量实现，以下规范确保各模块可被独立交付。

### 2.1 模块独立性

- 每个模块（`tmux_discovery`、`capture`、`event_parser` 等）可独立测试，不依赖其他模块
- 模块间通信仅通过 SQLite 或明确类型的函数参数，不使用共享全局状态
- 不从其他模块的内部实现导入，只依赖其公开接口（TypedDict / Pydantic 模型）

### 2.2 接口优先开发顺序

对每个模块按以下顺序实现：

```
1. 定义所有输入/输出的 TypedDict 或 Pydantic 模型
2. 将规范文档中的验收标准写成 pytest 测试桩（全部 failing）
3. 实现代码直到所有测试通过
4. 当前模块测试全绿后才进入下一个模块
```

### 2.3 防御性默认行为

- 所有与 tmux 交互的函数必须优雅处理「tmux 未运行」的情况（返回空，不抛出异常）
- 所有数据库写入必须包含 `try/except`，失败时记录日志但不中断采集流程
- 所有 Anthropic API 调用必须设置 `timeout=30`，失败重试最多 2 次
- WebSocket 广播在无客户端连接时必须立即返回，不阻塞

### 2.4 配置使用规范

代码中禁止出现魔法数字，所有阈值和可变参数通过 `config.loader.load_config()` 读取：

```python
# ✅ 正确
cfg = load_config()
threshold = cfg["thresholds"]["stuck_similarity"]

# ❌ 错误（魔法数字）
threshold = 0.92
```

## 3. Phase 1 实现顺序

以下顺序考虑了模块依赖关系，每一步均可独立验证：

| 步骤 | 任务 | 输出产物 |
|------|------|---------|
| 1 | `config/` 目录 — 加载器、路径工具、`ensure_dirs()` | `config/loader.py`、`config/paths.py` |
| 2 | `migrations/` — alembic 初始化，创建第 3 章所有表 | `migrations/versions/001_initial.py` |
| 3 | `daemon/tmux_discovery.py` — 附测试 | 通过 TC-DISC-001~004 |
| 4 | `daemon/capture.py` — 附测试（mock libtmux） | 通过 TC-CAP-001~004 |
| 5 | `daemon/event_parser.py` — 附测试（fixture 文本文件） | 通过 TC-PARSE-001~007 |
| 6 | `daemon/session_tracker.py` — 附测试（mock parser 事件） | 通过 TC-TRACK-001~005 |
| 7 | `daemon/attention_tracker.py` — 附测试 | 通过 TC-ATTN-001 |
| 8 | `daemon/main.py` — 编排以上模块，主轮询循环 | 可作为进程独立运行 |
| 9 | `server/db.py` + `server/models.py` | ORM 层就绪 |
| 10 | `server/routers/sessions.py` — 仅 GET 接口 | 通过 TC-API-001 |
| 11 | `server/ws.py` — WebSocket 广播 | 通过 TC-WS-001~003 |
| 12 | `server/routers/actions.py` — send + focus | 通过 TC-API-003~005 |
| 13 | `server/main.py` — 完整 FastAPI app | Server 可独立运行 |
| 14 | 前端基础 — SessionCard、StatusBadge、useWebSocket | Dashboard 可显示数据 |
| 15 | 前端交互 — QuickReply、内联编辑、发送确认 | 通过 TC-E2E-INPUT-001~004 |
| 16 | `launchd/` plist 文件 + README 安装说明 | 可一键安装到 macOS |

## 4. 开放问题与延迟决策

| # | 问题 | 当前默认假设 | 重新评估时机 |
|---|------|------------|------------|
| 1 | 多个 tmux server（`-L socket`）支持？ | Phase 1 仅支持默认 server | Phase 1 QA 阶段 |
| 2 | sqlite-vec → ChromaDB 迁移触发条件？ | session 数量超过 2000 时迁移 | Phase 2 评审 |
| 3 | Skill 草稿的格式规范？ | 遵循现有 `/mnt/skills/` 的 SKILL.md 格式 | Phase 3 启动前 |
| 4 | 各模型的 Token 单价表？ | 硬编码 claude-sonnet-4 定价，通过配置可覆盖 | Phase 1 构建时 |
| 5 | 移动端 / 平板 UI？ | 桌面优先，响应式布局延后 | Phase 1 发布后 |
| 6 | 多用户 / 团队模式？ | 单用户，本地私有数据 | Phase 3 之后 |
| 7 | context_window_pct 的模型上限表？ | 从 Anthropic API 的 model info 接口获取，本地缓存 | Phase 1 构建时 |

## 5. 文档维护规范

- 代码变更涉及接口契约（TypedDict 字段、API 路径）时，必须同步更新对应文档文件
- 新增 tmux 状态码时，同步更新 `03_数据模型.md` 的 Status Enum 表和 `05_Phase1_服务层与前端.md` 的 Badge 颜色映射
- 配置项变更（新增/删除/改名）时，必须同步更新 `02_多环境配置.md` 中的所有 `.toml` 示例和对比表
- 每个文档文件头部注明最后更新的 Phase 和日期
